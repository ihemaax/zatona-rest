# مراجعة احترافية شاملة للموقع (Web Audit)

تاريخ المراجعة: 2026-04-06

## منهجية المراجعة
- مراجعة بنية المشروع (Routes / Controllers / Views / Middleware).
- مراجعة تجربة المستخدم UX على مسار العميل (تصفح، سلة، Checkout، تتبع الطلب).
- مراجعة قابلية التوسع، الأمان، الأداء، SEO، وإمكانية الوصول.
- مراجعة الجودة الهندسية (تنظيم الكود، التكرار، الاختبارات).

---

## الملخص التنفيذي
الموقع غني وظيفيًا ويغطي دورة الطلب كاملة (منيو، سلة، Checkout، إدارة، توصيل)، لكن يوجد عدد واضح من التحسينات المطلوبة للوصول لمستوى Production احترافي عالي:

1) **تحسينات حرجة في SEO والـ Metadata** (لا توجد Meta description / OpenGraph بشكل قياسي في Layout الأساسي).
2) **اعتماد كبير على CSS/JS inline + CDN** بدل bundling/optimization، ما يؤثر على الأداء والـ maintainability.
3) **تكرارات وتباين في الصلاحيات والـ routing** تزيد احتمالية الأخطاء عند التوسع.
4) **صفحات معلوماتية (About/FAQ/Privacy/Contact) حالياً مبدئية** وتحتاج تعميق محتوى وعمليات تواصل فعلية.
5) **الاختبارات الحالية غير كافية** لتغطية السيناريوهات الحرجة (Checkout, Cart, Payment flow, i18n, security edge cases).

---

## نقاط القوة الحالية
- دعم عربي/إنجليزي مع تبديل اللغة وتفعيل اتجاه الصفحة حسب اللغة.
- تدفق طلب واضح: اختيار المنتج > السلة > نوع الاستلام > Checkout > نجاح الطلب.
- وجود نظام أدوار وصلاحيات للموظفين.
- وجود لوحة إدارة متقدمة نسبيًا (منتجات، فروع، طلبات، تقارير، توصيل).
- وجود اختبارات مبدئية لجزء من تدفق التوصيل.

---

## الفجوات الحالية (What’s Missing)

### 1) SEO وظهور الموقع في نتائج البحث (أولوية عالية)
**الملاحظات**
- Layout العميل يحتوي فقط على title و viewport و csrf بدون meta description أو OG tags أو canonical.
- لا يوجد تنظيم واضح لـ structured data (مثل LocalBusiness / Restaurant schema).

**الأثر**
- ظهور أضعف في البحث ومشاركة أقل جودة عند نشر الروابط على واتساب/فيسبوك/إكس.

**المطلوب**
- إضافة system موحد للـ SEO per-page (title, description, og:image, canonical, robots).
- توليد sitemap.xml وتهيئة robots.txt ديناميكي.
- إضافة JSON-LD للمطعم والفروع.

### 2) الأداء (Performance) (أولوية عالية)
**الملاحظات**
- وجود كتل CSS ضخمة inline داخل layouts والصفحات.
- تحميل مكتبات من CDN مباشرة في أكثر من موضع.
- استخدام صور خلفية كبيرة من مصادر خارجية بدون استراتيجية optimization واضحة.

**الأثر**
- بطء أول تحميل خصوصًا على الموبايل والاتصال المتوسط.
- صعوبة caching الفعال وتدهور Core Web Vitals.

**المطلوب**
- نقل الأنماط إلى ملفات CSS منظمة + استغلال Vite build/minify.
- تطبيق lazy-loading للصور واستخدام WebP/AVIF.
- preconnect/font-display وتحسين خط تحميل الخطوط.

### 3) تجربة المستخدم UX/CX (أولوية عالية)
**الملاحظات**
- صفحة Contact تنص أن نموذج التواصل قد يضاف لاحقًا (حاليًا لا يوجد قناة تواصل فورية داخل الموقع).
- صفحات المعلومات (About/Privacy/FAQ) مناسبة كبداية لكنها ليست كافية لعلامة تجارية تشغيلية فعلية.
- Checkout يعتمد على خريطة geolocation جيدة، لكن لا توجد رسائل/مساعدة متقدمة عند فشل geolocation أو ضعف الشبكة.

**الأثر**
- انخفاض الثقة وإحتمال drop-off قبل إتمام الطلب.

**المطلوب**
- إضافة نموذج تواصل (بريد/واتساب/تذكرة) مع SLA واضح.
- تطوير صفحة الأسئلة الشائعة لتشمل الإلغاء، الاسترجاع، أوقات التوصيل، الحد الأدنى.
- تحسين نسخ UX microcopy للسيناريوهات الفاشلة (Location denied / timeout / network).

### 4) الأمان والموثوقية (أولوية عالية)
**الملاحظات**
- تدفق إنشاء الطلب يمسك الاستثناءات برسالة عامة فقط بدون تسجيل خطأ واضح داخل الـ catch.
- يوجد تكرار routes للتقارير ما قد يسبب ارتباك صيانة مستقبلي.
- بعض صلاحيات العرض تعتمد على `is_admin` في الواجهة، بينما باقي النظام يتجه لنموذج role/permission؛ التوحيد مطلوب.

**الأثر**
- صعوبة تتبع الأعطال في الإنتاج.
- احتمال سلوك غير متوقع عند تعديل الصلاحيات لاحقًا.

**المطلوب**
- إضافة logging منظم للأخطاء مع correlation id للطلبات.
- إزالة التكرار في routes وتوحيد سياسة صلاحيات الواجهة مع الـ middleware.
- إضافة rate limiting للمسارات الحساسة (auth, cart mutations, checkout).

### 5) الجودة الهندسية والاختبارات (أولوية عالية)
**الملاحظات**
- وجود اختبارات قليلة مقارنة بحجم المنصة.
- لا يظهر وجود تغطية تلقائية لمعايير الجودة (linting/format/security scan) ضمن CI واضحة بالمشروع.

**الأثر**
- مخاطر regression أعلى عند كل تطوير.

**المطلوب**
- توسيع الاختبارات: Feature + integration لتدفقات الطلب كاملة.
- إضافة اختبارات صلاحيات roles/permissions edge cases.
- إضافة pipeline CI تشمل test + static analysis + style.

### 6) التحليلات وقياس النمو (أولوية متوسطة)
**الملاحظات**
- لا يوجد تتبع واضح لمؤشرات funnel (زيارة > إضافة للسلة > بدء checkout > نجاح الطلب).

**المطلوب**
- إضافة event tracking موحد (GA4/Matomo/Mixpanel).
- Dashboard KPIs: conversion rate, AOV, cart abandonment.

### 7) سهولة الوصول (Accessibility) (أولوية متوسطة)
**الملاحظات**
- الاعتماد الكبير على عناصر تصميمية مخصصة يتطلب تدقيق contrast / keyboard navigation / aria labels.

**المطلوب**
- إجراء audit WCAG 2.2 AA.
- دعم focus states واضحة ومراجعة التباين اللوني.

## فرص تطوير مباشرة (Quick Wins خلال 7–14 يوم)
1. بناء SEO partial موحد داخل layout.
2. استخراج CSS inline الأساسي إلى ملفات مجمعة عبر Vite.
3. إصلاح تكرار Route للتقارير.
4. إضافة logging داخل Checkout catch.
5. إطلاق صفحة Contact بنموذج فعلي + إرسال بريد.
6. إضافة 10–15 اختبار Feature لتدفقات Cart/Checkout/Permissions.

## خارطة طريق مقترحة (Roadmap)

### المرحلة 1 (أسبوعين)
- SEO foundations
- Performance quick wins
- Logging + route cleanup
- اختبار تدفق الطلب الأساسي

### المرحلة 2 (2–4 أسابيع)
- تحسين UX checkout/error states
- analytics funnel
- تطوير المحتوى الثابت والسياسات القانونية بشكل احترافي

### المرحلة 3 (4–8 أسابيع)
- تحسينات متقدمة: caching strategy, image pipeline, advanced search/filter, personalization
- تغطية اختبارية أوسع + CI quality gates

- تحسن LCP بنسبة 25%+
- خفض معدل التخلي عن السلة 15%+
- رفع معدل إكمال الطلب 10%+
- تقليل أخطاء الإنتاج المتكررة بنسبة 30%+

## ملاحظات تنفيذية
- التوصية الأساسية: البدء بـ **SEO + الأداء + الاختبارات** لأنها تعطي أعلى عائد مباشر على النمو والجودة.
- أي تطوير جديد في الواجهة يجب أن يمر على checklist موحد: Performance / Accessibility / Security / Analytics.
