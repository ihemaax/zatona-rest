# تقرير مراجعة أمنية (Static Review)

تاريخ المراجعة: 2026-04-07

## 1) ثغرة صلاحيات على مستوى لوحة الإدارة (Broken Access Control)

**الخطورة:** عالية

### الوصف
في `routes/admin.php` أغلب مسارات الإدارة محمية فقط بـ `auth` + `admin`، بينما التحقق الدقيق من الصلاحيات (`permission:*`) مفعّل فقط لأجزاء محدودة (مثل التقارير وإدارة الموظفين والكاشير/الدليفري في مجموعات محددة).

هذا يسمح لأي مستخدم "Staff" فعّال قادر على دخول لوحة الإدارة (مثل kitchen / delivery / branch_staff) بالوصول لوحدات حساسة مثل:
- إدارة الفروع/الأقسام/المنتجات.
- الإعدادات العامة.
- إدارة الكوبونات.

### الدليل
- تجميع المسارات تحت `middleware(['auth', 'admin'])` بدون `permission` لمعظم الوحدات في ملف المسارات.  
- وجود نظام صلاحيات جاهز (`hasPermission`, `canManageStaff`, `permission` middleware) لكن غير مطبق بشكل شامل على جميع الوحدات.

### الأثر
- تصعيد صلاحيات أفقي داخل لوحة الإدارة.
- تعديل بيانات تشغيلية حساسة بدون التفويض المناسب.

### التوصية
- إلزام كل وحدة Admin بصلاحية واضحة (مثال: `manage_products`, `manage_categories`, `manage_branches`, `manage_settings`, `manage_digital_menu`).
- إضافة `abort_unless` داخل كل Controller كخط دفاع ثانٍ.

## 2) مسارات الديمو الإدارية مكشوفة بدون تسجيل دخول

**الخطورة:** متوسطة

### الوصف
مسارات:
- `/demo/admin-dashboard`
- `/demo/admin/{path?}`

متاحة بدون أي Middleware توثيق/تفويض.

### الدليل
تعريف المسارات موجود خارج مجموعة `prefix('admin')->middleware(['auth', 'admin'])`.

### الأثر
- كشف هيكل النظام الإداري ومساراته لأي زائر.
- زيادة سطح الهجوم عبر جمع معلومات عن الوحدات الداخلية (Reconnaissance).

### التوصية
- قصر الديمو على بيئة التطوير فقط (feature flag / `app()->environment('local')`).
- أو إضافة مصادقة حتى للديمو العام.

---

## 3) رفع صور SVG في الإعدادات بدون سياسة تعقيم واضحة

**الخطورة:** متوسطة

### الوصف
`SettingController` يسمح برفع شعار بصيغة `svg` ضمن:
- `logo => mimes:jpg,jpeg,png,webp,svg`

الـ SVG قد يحتوي JavaScript أو محتوى نشط إذا لم تتم معالجته/تعقيمه قبل العرض.

### الدليل
التحقق يسمح بالـ SVG ثم يتم تخزين الملف مباشرة في `public` disk.

### الأثر
- احتمالية Stored XSS عند عرض الشعار في الواجهة حسب طريقة تضمينه (`img/object/inline`).

### التوصية
- منع SVG في الرفع العام، أو
- تطبيق SVG sanitizer موثوق قبل الحفظ،
- وإجبار العرض عبر `<img>` مع تقييد صارم لرؤوس الأمان (CSP + no-sniff).

## 4) نقص Rate Limit لمسار المساعد الذكي (AI)

**الخطورة:** متوسطة

### الوصف
مسار `/admin/ai-assistant/ask` لا يظهر عليه `throttle` رغم أنه يستدعي API خارجي مدفوع (Groq).

### الأثر
- استنزاف مالي (API abuse).
- ضغط زائد على البنية التحتية وتأخير الخدمة.

### التوصية
- إضافة Rate Limit صارم لكل مستخدم/دقيقة ولكل ساعة.
- إضافة Quota يومي لكل دور وظيفي إن لزم.

---

## 5) تسريب تفاصيل أخطاء مزود AI للمستخدم النهائي

**الخطورة:** منخفضة إلى متوسطة

### الوصف
`GeminiService::ask` يعيد نص الخطأ الداخلي للمستخدم مباشرة:
- `Groq مش شغال حاليًا. الخطأ: ...`
- ويتضمن حالة HTTP و`response body` من مزود خارجي.

### الأثر
- كشف معلومات تشغيلية داخلية (صيغة الطلب/تفاصيل الفشل).
- يفيد المهاجم في تحسين محاولات الاستغلال.

### التوصية
- إرجاع رسالة عامة للمستخدم.
- تسجيل التفاصيل الكاملة في logs فقط (بدون إظهارها في UI).

---

## ملاحظات عامة
- لا توجد مؤشرات مباشرة في هذا الفحص على SQL Injection تقليدي داخل الاستعلامات الظاهرة (الاستخدام كان غالبًا Parameterized).
- التحقق من CSRF يعتمد على Laravel Web middleware (جيد مبدئيًا).
- يفضّل إضافة اختبارات أمنية تلقائية (authorization tests) لكل مسار إداري.
