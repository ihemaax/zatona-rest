# مراجعة أمنية شاملة للموقع (2026-04-07)

## نطاق المراجعة
- مراجعة إعدادات الجلسات والكوكيز والهيدرز الأمنية.
- مراجعة مسارات الويب والصلاحيات والـ middleware.
- مراجعة نقاط إدخال المستخدم (checkout/auth/uploads) ومخاطر XSS/IDOR/SQLi.
- مراجعة مخاطر النشر والتشغيل الظاهرة داخل المستودع.

## نقاط القوة
1. **حماية صلاحيات الإدارة موجودة** عبر `auth + admin` مع middleware صلاحيات تفصيلية.
2. **تحديد المعدل (Rate Limiting)** مفعّل لمسارات مهمة مثل السلة، checkout، OTP، والمساعد الذكي.
3. **تطبيق هيدرز أمنية أساسية** مثل `X-Frame-Options`, `nosniff`, `Referrer-Policy`, `CSP`, وHSTS على HTTPS.
4. **حماية طلب نجاح الطلبات (Order Success)** موجودة بربط المستخدم أو `guest_token` للضيوف.
5. **تشفير/تقوية الجلسة والكوكيز** تم تشديدها (Secure + HttpOnly + SameSite + Session encryption + fingerprint rotation).

## نقاط الضعف والثغرات المكتشفة

### حرجة (Critical)
1. **وجود نسخ قواعد بيانات داخل المستودع** (`backup.sql`, `all_databases_backup.sql`) يمثل خطر تسريب بيانات شديد جدًا عند أي خطأ نشر أو وصول غير مصرح.
2. **مسارات ديمو للإدارة متاحة بدون `auth`** عبر `/demo/admin-dashboard` و`/demo/admin/{path?}`؛ قد لا تكشف بيانات تشغيل مباشرة لكنها تكشف بنية اللوحة ومساراتها وتزيد سطح الهجوم.

### عالية (High)
3. **رفع SVG مسموح به للشعار** في إعدادات الإدارة. ملفات SVG قد تحمل JavaScript/حمولات XSS إذا قُدمت مباشرة من public storage بدون تعقيم صارم.
4. **سياسة CSP ما زالت تسمح `unsafe-inline` للـ scripts/styles** مما يقلل فعالية CSP ضد XSS.

### متوسطة (Medium)
5. **حماية الجلسة بالـ fingerprint تعتمد على IP + User-Agent** وقد تسبب تسجيل خروج متكرر للمستخدمين خلف شبكات متغيرة (Mobile/CGNAT)، وتحتاج مراقبة توازن الأمن/سهولة الاستخدام.
6. **منطق Frontend ما زال ظاهرًا للعميل** (وهو متوقع تقنيًا). المنع الكامل من DevTools غير ممكن؛ الحل الصحيح هو عدم وضع أسرار أو قواعد حساسة في الواجهة.

### منخفضة (Low)
7. **رسائل أخطاء/تحذيرات Console في الإنتاج** تم تقليلها بالبناء، لكن لا تزال رؤية DOM/JS من المتصفح ممكنة بطبيعة الويب.

## توصيات تنفيذية (مرتبة بالأولوية)
1. **إزالة أي dump/backup DB من المستودع فورًا** وتدوير جميع كلمات المرور/المفاتيح إذا كانت البيانات حقيقية.
2. **قفل مسارات الديمو** بواحد من:
   - حذفها من الإنتاج نهائيًا، أو
   - تقييدها بـ `auth + admin + env=local/staging`.
3. **منع SVG uploads أو تعقيمها تعقيمًا قويًا** (sanitize pipeline) قبل الحفظ/العرض.
4. **الانتقال إلى CSP بدون `unsafe-inline`** عبر nonce/hash ونقل السكربتات inline إلى ملفات bundle.
5. **تفعيل إعدادات إنتاج إلزامية**: `APP_DEBUG=false`, `APP_ENV=production`, TLS فقط، وقواعد web server لمنع الوصول إلى ملفات حساسة (`.env`, `*.sql`, `.git`).
6. **تقوية الطبقة الشبكية**: WAF + fail2ban + مراقبة محاولات brute force + rate limits على تسجيل الدخول.
7. **إجراء Pentest خفيف دوري** + فحوص تلقائية (SAST/Dependency audit) ضمن CI.

## الخلاصة
- الموقع لديه أساس جيد (صلاحيات + rate limit + هيدرز + جلسات محمية)،
- لكن توجد **نقطتان خطيرتان جدًا** يجب التعامل معهما فورًا: **ملفات النسخ الاحتياطي لقواعد البيانات داخل المشروع** و**مسارات الديمو المفتوحة بدون توثيق**.
